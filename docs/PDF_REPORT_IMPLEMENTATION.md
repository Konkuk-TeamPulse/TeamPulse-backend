# TeamPulse PDF 리포트 구현 정리

이 문서는 PDF 리포트 서식 보강 내용을 Notion이나 제안서 구현 설명에 옮길 때 참고하기 위한 백엔드 기준 정리입니다.

## 구현 방식

- `ProjectApiController`의 PDF 다운로드 API는 기존 path를 유지합니다.
- 컨트롤러는 `ReportPdfService`에 프로젝트 상태와 리포트 메타데이터를 넘깁니다.
- `ReportPdfService`는 도메인 데이터를 PDF 전용 View Model로 집계한 뒤 HTML/CSS 템플릿을 생성합니다.
- HTML은 `openhtmltopdf-pdfbox`로 PDF 바이트 배열로 렌더링합니다.
- 한글 폰트는 `app.report.pdf.font-path` 설정값을 우선 사용하고, 없으면 OS 기본 한글 폰트를 탐색합니다.

## 포함 섹션

1. 표지 및 핵심 요약
2. 프로젝트 개요
3. 전체 진행 요약
4. 태스크 진행 현황
5. 팀원별 작업 현황
6. 지연 및 병목 확인 항목
7. 회의 및 의사결정 요약
8. 리스크 확인 항목
9. 다음 액션
10. 활동 로그 요약

## 계산 기준

- 기준일은 리포트 기간의 종료일입니다. 기간 파싱이 실패하면 생성일 기준으로 처리합니다.
- 완료율은 `DONE / 전체 태스크 수`로 계산합니다.
- 태스크가 0개면 완료율을 0%로 표시하지 않고 `계산 불가`로 표시합니다.
- 일정 확인 필요 태스크는 `DONE`이 아니고 마감일이 기준일보다 이전인 태스크입니다.
- 마감 임박 태스크는 `DONE`이 아니고 기준일로부터 3일 이내 마감인 태스크입니다.
- 병목 확인 태스크는 `DONE`이 아니고 blocker가 등록된 태스크입니다.
- 팀원별 작업 현황은 태스크 담당자 이름과 팀원 이름을 기준으로 집계합니다.
- 다음 액션은 회의 액션 아이템, 리스크 제안 액션, 확인 필요 태스크 순서로 구성합니다.

## 표현 원칙

- 실제 데이터에 없는 내용은 생성하지 않습니다.
- "문제 발생", "위험 확정", "실패 가능성" 같은 단정 표현은 쓰지 않습니다.
- "확인 필요", "진행 상태 확인", "추가 점검"처럼 보수적인 문구를 사용합니다.
- 리스크 심각도는 저장된 값을 그대로 라벨만 바꿔 표시합니다.

## API 영향

요청 path와 응답 방식은 변경되지 않았습니다.

- `POST /api/projects/{projectId}/reports`
- `GET /api/projects/{projectId}/reports`
- `GET /api/projects/{projectId}/reports/{reportId}/download`
- `GET /api/reports/{reportId}/download`

프론트는 기존처럼 PDF 파일 다운로드만 처리하면 됩니다.

## 남은 고도화

- 배포 서버에 한글 폰트가 없을 경우 `app.report.pdf.font-path` 환경 설정이 필요합니다.
- 대형 프로젝트에서는 상세 태스크/회의/로그를 부록으로 분리하는 정책이 추가로 필요합니다.
- 기간별 추세 그래프, 태스크 상태 변경 이력, 팀원별 workload 균형 시각화는 아직 구현하지 않았습니다.
