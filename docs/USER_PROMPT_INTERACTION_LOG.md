# 사용자 프롬프트 및 상호작용 로그

이 문서는 TeamPulse 백엔드 작업 과정에서 사용자가 어떤 방식으로 요구사항을 전달했고, 대화가 어떤 방향으로 전개되었는지 정리한 기록입니다.

원문 전체를 그대로 저장하는 목적이 아니라, 이후 GPT Pro, Claude Code, Codex, Cursor 등에 작업을 맡길 때 참고할 수 있도록 **요구사항 전달 방식, 의사결정 흐름, 프롬프트 패턴**을 정리했습니다.

## 1. 전체 흐름 요약

이번 대화는 TeamPulse 프로젝트를 기준으로 다음 흐름으로 진행되었습니다.

1. 문서와 깃 최신 버전 기준으로 DB/API 상태 확인
2. 로컬을 원격 develop 기준으로 동기화
3. MySQL 기반 DB 연결과 실제 API 동작 검증
4. API 변경사항 문서화 및 프론트 담당자 전달 문서 작성
5. 초대 링크, 로그인 토큰, 권한 흐름 검토
6. 제품 출시 전 필수 보완 항목 구현
7. PDF 리포트 산출물 고도화
8. 실제 더미 데이터 삽입 후 PDF 다운로드 및 보완점 검토

중간에 사용자는 “깃에 올리지 말고 로컬에서 움직이자”, “프론트는 담당자가 있으니 백엔드만 잡자”, “실행하지 말고 먼저 검토하자”처럼 작업 범위를 계속 조정했습니다.

## 2. 주요 요구사항 로그

### 2.1 DB 이슈 인식 및 문서 기반 계획 요청

카카오톡 대화에서 DB 사용 방식이 설계 문서와 다르면 감점될 수 있다는 우려가 나왔고, 사용자는 관련 PDF 문서를 읽고 깃 최신 버전을 기준으로 DB 연결 계획을 세워달라고 요청했습니다.

핵심 요구:

- 요구사항/기획 문서 두 개 정독
- 깃 최신 버전 기준으로 현재 구현 상태 확인
- 로컬이 오래됐으면 최신 develop 기준으로 덮기
- DB 연결 목표 수립
- 실제 구현과 설계 문서가 어긋날 지점 찾기
- 10년차 백엔드 개발자처럼 구현 방향 고민

이 단계에서 사용자는 단순 코드 수정이 아니라, 평가/문서/구현 일치 여부까지 함께 검토하길 요구했습니다.

### 2.2 로컬 최신화와 DB 구현

사용자는 로컬을 원격 develop 기준으로 덮고, 필요한 의존성 설치 후 DB 구현과 실행 검증을 요청했습니다.

핵심 요구:

- 로컬 변경보다 깃 최신 기준 우선
- 필요한 도구와 의존성 설치
- MySQL 기반 DB 연결 구현
- 실행 검증까지 수행
- API 호출 구조가 바뀌면 별도 문서로 정리
- Notion 수정이 필요할 수 있으니 변경사항을 찾기 쉽게 남기기

이때 사용자는 `API_CHANGES.md`처럼 이후 문서 담당자가 볼 수 있는 변경 로그를 요구했습니다.

### 2.3 실제 사용자 흐름 기반 검증 요청

DB 구현 후 사용자는 단순 테스트 통과가 아니라, 실제 사용자가 앱 기능을 쓰는 것처럼 데이터를 넣고 흐름을 검증해달라고 요청했습니다.

핵심 요구:

- 실제 데이터를 집어넣기
- 앱 기능을 사용자처럼 사용하기
- 가능한 루트지만 구현이 미흡한 부분 찾기
- 프로젝트 ID가 늘어나도 잘 동작하는지 확인
- DB도 함께 연결해서 유지보수 가능한 방향으로 개선

이 요청을 통해 단일 `projectId=1` 가정, 멀티 프로젝트 격리, 초대 수락 후 프로젝트 목록, 리포트 생성/다운로드 흐름 등이 점검되었습니다.

### 2.4 API 변경사항 문서화와 팀원 전달

DB/API 변경 후 사용자는 프론트 담당자와 Notion 담당자가 이해할 수 있게 수정사항을 쉽게 정리해달라고 요청했습니다.

핵심 요구:

- 컴공 3학년이 봐도 이해할 수 있게 작성
- 무엇이 변경되었는지 명확하게 정리
- 프론트 담당자가 연결 확인할 수 있게 API path 목록 제공
- 카톡 공지 문구 작성
- Notion에 어떤 내용을 반영해야 하는지 검토

결과적으로 다음 문서들이 생성/보강되었습니다.

- `docs/API_CHANGES.md`
- `docs/DB_AUTH_HANDOFF.md`
- `docs/FRONTEND_BACKEND_HANDOFF.md`
- `docs/PDF_REPORT_IMPLEMENTATION.md`

### 2.5 초대 링크와 로그인 토큰 관련 질문

프론트 담당자가 “초대 링크로 들어오면 어떤 사용자인지 어떻게 파악하느냐”고 질문했고, 사용자는 백엔드 입장에서 어떻게 답해야 하는지 물었습니다.

핵심 논점:

- 비로그인 상태에서는 서버가 사용자를 알 수 없음
- 초대 조회는 공개 가능
- 초대 수락은 로그인 토큰 필요
- 로그인한 상태라면 Authorization token으로 서버가 사용자 식별
- request body에 userId를 넘기기보다 로그인 토큰 기반이 안전함

이 단계에서 사용자는 팀원에게 보낼 답변 문장까지 요청했습니다.

### 2.6 제품 출시 수준 보완 요구

사용자는 “제품 출시 가능?”이라고 물었고, 이후 출시 전 필수 보완 항목을 순차적으로 구현하라고 지시했습니다.

핵심 보완:

- JWT access token 도입
- refresh token rotation
- token hash 저장
- 로그인 실패 횟수 제한
- 이메일 존재 여부 노출 방지
- LEADER/MEMBER 권한 분리
- 초대 수락자는 항상 MEMBER
- 운영 profile 분리
- `/health`, `/v3/api-docs` 공개 범위 정리
- PDF 리포트 완성도 향상

이 구간에서 백엔드는 MVP 수준에서 제품화 1차 보완 수준으로 올라갔습니다.

### 2.7 Git 작업에 대한 범위 조정

중간에 사용자는 Git push 여부에 대해 강하게 확인했습니다.

핵심 요구:

- 사용자가 명시하지 않은 push 금지
- “내가 하라는 것만 해”
- 일단 로컬에서만 움직이기
- 팀 규칙에 맞는지 검토가 필요할 때만 Git 작업 고려

이후 작업은 로컬 구현과 검증 중심으로 진행되었습니다.

### 2.8 프론트 담당자 분리와 백엔드 handoff

사용자는 “프론트는 담당이 있으니 따로 문서 정리해서 건네주자”고 했습니다.

핵심 요구:

- 프론트 코드는 직접 수정하지 않기
- 백엔드 변경사항만 정리
- 프론트 담당자가 필요한 API 수정사항, 토큰 처리, 권한 처리, PDF 다운로드 흐름을 문서로 전달

이 요구에 따라 `FRONTEND_BACKEND_HANDOFF.md`가 작성되었습니다.

### 2.9 PDF 리포트 고도화

사용자는 GPT Pro에게 받은 PDF 리포트 설계안을 붙여 넣고, 백엔드에서 실제 데이터 기반으로 PDF를 고도화하자고 요청했습니다.

핵심 요구:

- 실제 데이터에 없는 내용 생성 금지
- 위험/문제 확정 표현 금지
- 교수님 제출용처럼 차분한 레이아웃
- 표지, 프로젝트 개요, 전체 진행 요약, 태스크 현황, 팀원별 작업, 지연/병목, 회의, 리스크, 다음 액션, 활동 로그 포함
- 더미 데이터를 직접 넣고 PDF를 받아서 보완점 검토
- 프론트는 두고 백엔드 PDF만 잡기

실제 수행:

- 더미 프로젝트 생성
- 팀원 4명 생성
- 태스크 9개 생성
- 회의 2개 생성
- 액션 아이템 4개 생성
- 의존관계 추가
- 리스크 자동 생성 확인
- PDF 다운로드
- 텍스트 추출과 페이지 이미지 렌더링으로 검수
- 카드형 레이아웃, 한국어 활동 로그, 완화된 리스크 문구로 개선

## 3. 사용자의 프롬프트 방식 분석

### 3.1 실제 팀 상황을 함께 제공

사용자는 단순히 “DB 연결해줘”라고 하지 않고, 팀원 카카오톡 대화, 교수님 평가 우려, Notion 문서, 프론트 담당자 질문을 함께 제공했습니다.

이 덕분에 다음 내용을 함께 판단할 수 있었습니다.

- 팀원이 무엇을 헷갈리는지
- 문서와 구현이 어디서 어긋날 수 있는지
- 프론트 담당자에게 무엇을 전달해야 하는지
- 평가에서 어떤 부분이 위험한지

### 3.2 중간에 목표를 강하게 재정렬

사용자는 작업 방향이 어긋날 가능성이 있을 때 바로 정정했습니다.

대표 패턴:

- “실행하지 말고 먼저 검토해.”
- “프론트는 담당이 있으니 백엔드만.”
- “깃에 올리지 말고 로컬에서 움직이자.”
- “내가 하라는 것만 해.”
- “문서 반영은 내가 나중에 할게.”

이 방식은 작업 범위가 커지는 것을 막는 데 효과적이었습니다.

### 3.3 “10년차 시니어”처럼 품질 기준을 명시

사용자는 여러 번 “10년차 백엔드 개발자처럼”, “완제품 수준”, “시판 제품처럼” 같은 표현으로 품질 기준을 높였습니다.

이 표현은 작업자의 판단 기준을 단순 MVP 데모 수준에서 운영/출시 관점으로 끌어올리는 효과가 있었습니다.

다만 이런 요구는 구현 범위가 크게 늘어나므로, 각 단계마다 “이번에는 어디까지 할지”를 함께 정하면 더 안정적입니다.

### 3.4 검토와 실행을 분리

사용자는 중요한 변경 전에는 검토 단계를 요구했습니다.

이 패턴은 특히 다음 작업에서 유효했습니다.

- Git push 여부
- DB overwrite
- Notion 반영
- API path 변경
- 프론트 담당자에게 전달할 내용
- PDF 고도화 방향

### 3.5 실제 데이터 기반 검증을 요구

사용자는 단순한 단위 테스트보다 “실제로 데이터를 넣고 앱의 기능들을 사용자처럼 사용해보라”고 요구했습니다.

이 방식은 다음 문제를 찾는 데 효과적이었습니다.

- projectId 고정 문제
- 초대 수락 후 role 처리
- 멤버 권한 문제
- PDF 리포트가 실제로 풍부한 데이터를 담는지
- 문서에 쓴 내용과 실제 API 동작이 맞는지

## 4. 프롬프트 개선을 위한 권장 템플릿

앞으로 비슷한 작업을 맡길 때는 아래 구조가 가장 안정적입니다.

```text
목표:
- 이번 작업의 최종 목표를 한 문장으로 작성

현재 상태:
- 로컬/깃/배포 중 어느 기준인지
- 이미 구현된 것
- 문제가 되는 API/파일/문서

이번에 할 일:
1. 반드시 수정할 항목
2. 검토만 할 항목
3. 건드리면 안 되는 항목

수용 기준:
- 어떤 동작이 되면 성공인지
- 어떤 테스트/API 호출을 통과해야 하는지
- 문서화가 필요한지

주의:
- git push 금지/허용
- 프론트 수정 금지/허용
- Notion 직접 수정 금지/허용
- 실행 전 검토 요청 여부
```

- “문서 두 개를 읽고 깃 최신 버전을 바탕으로 계획 짜봐.”
- “로컬을 최신으로 덮어버려.”
- “필요한 것들부터 다운 받고 실행 검증까지 해.”
- “API 호출 구조가 달라지면 따로 문서로 정리해.”
- “컴공 3학년이 봐도 알아먹기 쉽게.”
- “프론트는 담당이 있으니 백엔드만 잘 잡자.”
- “실제로 데이터를 집어넣고 사용자처럼 사용해봐.”
- “깃에 올리지 말고 로컬에서 움직이자.”
- “더미 데이터를 직접 넣고 PDF까지 받아와서 보완점을 검토해.”

## 6. 앞으로 남길 만한 작업 로그 기준

작업이 커질 때는 매번 아래 네 가지를 남기면 좋습니다.

1. 변경 파일
2. 변경 이유
3. 프론트/문서 담당자 영향
4. 검증한 명령/API/결과

이번 작업에서는 이 기준에 맞춰 API 변경 문서, DB/Auth handoff, Frontend handoff, PDF 구현 문서를 분리했습니다.
